# Flowmeter

## Documentation

The configuration of the documentation (and thus the creation of related targets) are controlled by the `ENABLE_DOCS` option.
Set it to:

- `ON`: to configure the documentation
- `OFF`: to not configure it

The default value is `ON`.

The documentation is generated in html format using [Sphinx](https://www.sphinx-doc.org/en/master/).
For this, a CMake target is added : `sphinx`

Its execution will cause the creation of a sub-folder `docs/build/html`.
Simply open the `index.html` file with your favorite web browser to view it.

The current configuration of Sphinx allows:

- Write documentation in [reStructuredText (RST)](https://www.sphinx-doc.org/en/master/usage/restructuredtext/basics.html) format
- Include UML diagrams in [PlantUML](https://plantuml.com) format
- Include [Doxygen](https://www.doxygen.nl) documentation into the RST files.

### Requirements

#### Sphinx

The generation of the html documentation requires [Python 3](https://www.python.org/about/gettingstarted/) and some additional packages using [pip](https://docs.python.org/3.9/installing/index.html).

To install the required package, run the following command:

```bash
pip install -r docs/requirements.txt
```

#### Doxygen

The installation of Doxygen is optional, but its absence can block the operation of certain functions, notably the inclusion of the documentation present in the source code directly in the documentation generated by Sphinx.

#### PlantUML

The installation of PlantUML is optional, but its absence can block the operation of certain functions, notably the inclusion of the diagram directly in the documentation generated by Sphinx.

##### PLantUML Server

If you don't have plantuml installed, you can always use a docker image setting up a server.

###### Install the official image

```bash
docker pull plantuml/plantuml-server:latest
```

###### Create a container

```bash
docker container create -p 8080:8080 -v "<absolute_path>/config.txt":"/tmp/config.txt" -e PLANTUML_CONFIG_FILE="/tmp/config.txt" --name pluml plantuml/plantuml-server
```

Where `<absolute_path>`, is the absolute path to the folder containing the `config.txt` file. Its content is automatically added to all diagrams.

###### Run the container

```bash
docker container start pluml
```

You should be able to see the PlantUML server at <http://localhost:8080>

It is not necessary to repeat the two previous steps each time.

##### Usage

Install the [VsCode Extension for PlantUML](https://marketplace.visualstudio.com/items?itemName=jebbs.plantuml) (first one when typing plantuml)

Head to the settings of the extension by clicking the gear icon on the right of the extension

In the settings set:

- the output format to svg
- PlantUml : Render -> select PlantUML Server
- the server to <http://localhost:8080>
- Plantuml : Url Format : select svg

You can also directly copy and paste the following lines into the `.vscode/settings.json` file of your workspace:

```json
{
    "plantuml.server": "http://localhost:8080",
    "plantuml.render": "PlantUMLServer",
    "plantuml.exportFormat": "svg"
}
```

You should be able to open a puml file and hit the render button top right to see the diagram
You can find the documentation about the syntax of PlantUML [here](https://plantuml.com/guide)

> If you are not using the PlantUML server, the `plantuml.render` parameter must remain `"Local"`, and you must add these lines to the parameter PlantUML: Jar Args:
>
> ```json
> {
>     "plantuml.jarArgs": [
>         "-config",
>         "<absolute_path>/config.txt"
>     ]
> }
> ```
>
> The content of the `config.txt` file is automatically included in all plantuml files by sphinx thanks to the `-config` option.

### Write documentation

#### Add PlantUML diagram

To add it is possible to do in different ways using the directive `.. uml::`.
See this [link](https://github.com/sphinx-contrib/plantuml/#usage) to know how to use it.

It is possible to use the [PlantUML preprocessor](https://plantuml.com/preprocessing) to include other diagrams in a diagram.

#### Add Doxygen documentation

The addition of documentation from the project's source code is done using different directives.
See this [link](https://breathe.readthedocs.io/en/latest/directives.html#directives) to know how to use them.

For your module to be recognized by Doxygen, you must give it the target used for this module:

```cmake
if(COMMAND reference_target_for_analysis)
    reference_target_for_docs(<your_target>)
endif(COMMAND reference_target_for_analysis)
```

> Note
> The use of ``if`` and ``endif`` is optional.

A ``doxygen-<your_target>`` target is then created, it will generate your doxygen in the ``docs/build/doxygen/<your_target>`` folder.

target ``docs`` is also created, it allows to launch all Doxygen targets and to generate the Sphinx documentation.

##### Add your project

The `conf.py` file in the `docs` folder is used to configure the [breathe](https://breathe.readthedocs.io/en/latest/index.html) package to integrate the Doxygen documentation.
To make your project documentation available (lib, executable, ...) you have to edit some variables in this file.

###### [breathe_projects_source](https://breathe.readthedocs.io/en/latest/directives.html#confval-breathe_projects_source)

The value contained in this dictionary is used by the `autodoxygen` directives.
So in order to use them for your project you have to fill in the path to your files starting from the __obsw-dev__ root folder (relative path from the file.

> __Example:__
>
> ```python
> breathe_projects_source = {
>     "example": [
>         os.path.abspath("../<path_to_my_project>"),
>         [
>             "<my_subfolder>",
>         ],
>     ]
> }
> ```

Replace `<path_to_my_project>` by the relative path from the file __README.md__.
Replace `<my_subfolder>` by the relative path of the sub-folders of your project, from the root of your project (from `<path_to_my_project>`).

The `example` key is then used by the `project` option of the `autodoxygen` directives.

## Analysis

The configuration of the analysis targets are controlled by the `ENABLE_ANALYSIS` option.
Set it to:

- `ON`: to configure the analysis target
- `OFF`: to not configure it

The default value is `ON`.

### Targets

Each project is referenced for analysis using the `reference_target_for_analysis()` function.
This will then create a target `clangformat-<target>` and `cppcheck-<target>`.

> __Example:__
>
> ```cmake
> add_executable(hello hello_world.c)
> reference_target_for_analysis(hello)
> ```
>
> A target `cppcheck-hello` and `clangformat-hello` is created.

In addition to this, a target `clangformat-all` and `cppcheck-all` is created.
This one has the particularity to execute the other target `clangformat-<target>` and `cppcheck-<target>`.

> __Example:__
>
> ```cmake
> add_executable(hello hello_world.c)
> reference_target_for_analysis(hello)
>
> add_executable(goodbye goodbye.c)
> reference_target_for_analysis(goodbye)
> ```
>
> A target `cppcheck-hello` and `clangformat-hello` is created.
> A target `cppcheck-goodbye` and `clangformat-goodbye` is created.
>
> A target `cppcheck-all` and run the targets `cppcheck-hello` and `cppcheck-goodbye`
> A target `clangformat-all` and run the targets `clangformat-hello` and `clangformat-goodbye`

### CppCheck

Part of the static analysis of the code is done using the [CppCheck](http://cppcheck.net/) tool.
Cppcheck covers almost all the [MISRA C](https://www.misra.org.uk/misra-c/) 2012 rules. Including the amendments.

It is still possible to ignore certain rules of the tool.

#### Inline suppress

One way to prevent a certain error is by using inline suppression. By following the format of `cppcheck-suppress <ERROR:ID>` CppCheck will recognize this, and will not print out the error.

> __Example:__
>
> ```C
> int foo()
> {
>     arr[10];
>     // cppcheck-suppress arrayIndexOutOfBounds
>     arr[15] = 1;
> }
> ```

To apply this rule to all files, simply add the option `--suppress=<ERROR_ID>` it in the command line. Example:

The suppression of a MISRA rule can also be indicated in the `analysis/cppcheck/misra.json` file.

> __Example:__
>
> Here, the MISRA rules `17.3` and `21.12` are ignored:
>
> ```json
> {
>     "script": "misra.py",
>     "args": [
>         "--rule-texts=misra.txt",
>         "--suppress-rules 17.3,21.12"
>     ]
> }
> ```

It is possible to put all the rules to be ignored in a file.
See the option `--suppressions-list=<file>` or `--suppress-xml=<file>p`

### ClangFormat

[ClangFormat](https://clang.llvm.org/docs/ClangFormat.html) is a tool to format C/C++/… code according to a set of rules and heuristics.

The [style](https://clang.llvm.org/docs/ClangFormatStyleOptions.html) to follow is defined in the `.clang-format` file.
